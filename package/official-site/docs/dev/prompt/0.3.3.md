总之：将删除、新增、更新照片的操作都封装成函数，所有地方都调用这个函数，避免直接操作数据库。

## 重复照片清理功能

计算重复照片的md5值是否相同，相同则认为是重复照片。

1. package\website\src\views\toolbox 新增清理重复照片的页面，与package\website\src\views\toolbox\SimilarPhotoCleanup.vue类似
2. package\server\app\db\models\photo.py 新增一个字段，用于存储照片的md5值
3. 前端触发扫描重复照片的按钮，点击后调用后端接口，新建一个任务，用于扫描重复照片，与扫描相似照片类似
   1. 后端扫描照片，计算没有md5值的照片的md5值，将md5值存储到照片表中
   2. 前端可以查看扫描进度，扫描完成后，显示重复照片的列表
   3. 用户可以在列表中选择要删除的重复照片，点击删除按钮，删除选中的照片照片

## Why

用户相册中可能存在大量完全相同的照片（例如多次上传、备份产生）。通过计算照片的MD5值，可以精确找出这些重复照片，并提供清理功能，帮助用户节省存储空间。

## What Changes

- 在后端 `Photo` 数据库模型中增加 `md5` 字段，用于持久化存储照片文件的MD5哈希值。
- 新增一个后台异步任务（Task），专门用于扫描尚未计算过MD5的照片并更新其MD5值，然后找出MD5值相同的重复照片分组。
- 提供相关的API接口，用于触发重复照片扫描任务、查询任务进度、获取重复照片列表、以及删除选定的重复照片。
- 在前端 `package/website/src/views/toolbox` 目录下新增 `DuplicatePhotoCleanup.vue` 页面（参考 `SimilarPhotoCleanup.vue`），提供完整的用户交互界面。

- 系统需要在照片模型中存储MD5值（字符串），以支持高效的完全相同照片检测。

### Requirement: 扫描重复照片任务

- 系统需要提供一个接口启动重复照片扫描任务。

## 任务处理优化

采用工厂方法模式和策略模式统一处理任务，每个任务类型对应一个策略类，策略类负责处理具体的任务逻辑。

1. 每个策略类自己负责处理自己的任务，不依赖其他策略类。策略类的实例化和调用在任务处理模块里，根据任务类型动态选择对应的策略类。
2. 现在任务处理是单个任务，考虑批量处理任务，以减少iO请求，提高处理效率。
3. 每个策略类要实现一个任务处理完成后的回调函数，用于处理任务处理完成后的逻辑。例如：package\server\app\service\task_worker.py的_flush_results里处理完photos_to_create之后要新增一些其他任务，这部分要在具体的策略类里实现，不再依赖任务处理模块。
4. 考虑使用线程池或者异步任务加快处理速度，以提高任务处理效率。

## 任务分配优化

package\server\app\service\task_worker.py 目前是通过_fetch_and_dispatch_tasks_sync函数从数据库里读取任务，取一部分拿出来处理，把数据库当做了任务队列。其实这些任务可以并发处理，例如CPU任务（构建缩略图、提取元数据等），iO任务（face、classify、ocr等，大量时间用在等待AI微服务返回结果），AI任务（也是大量时间都在等待外部大模型输出结果）等，这些任务可以并发处理，以提高处理效率。

所以，可以在package\server\app\service\task_worker.py新增多个任务队列，每个队列负责处理一种任务类型。
1. CPU任务队列：负责处理CPU任务，例如构建缩略图、提取元数据等
2. iO任务队列：负责处理iO任务，例如face、classify、ocr等
3. AI任务队列：负责处理AI任务，例如visual_description等

任务队列采用优先队列，优先级高的先处理，优先级低的后处理
_fetch_and_dispatch_tasks_sync函数从这些队列里面取任务，然后再执行对应的策略类。所以还需要在package\server\app\service\task_worker.py里新增一个任务队列管理模块，负责管理这些队列。

## 任务批处理优化

现有的AI微服务都支持批量处理任务，以减少iO请求，提高处理效率。而当前的任务处理模块是单个图片，考虑批量处理任务，以减少iO请求，提高处理效率。所以这些任务队列里每个任务可以包含多个子任务，分配发送给AI微服务进行处理，以减少iO请求次数和上下文切换次数，提高处理效率。

# 分布式缓存热点数据一致性检查方案

## 整体思路分析

这个问题的核心是：**在分布式缓存集群中，同一份热点数据（如被高频访问的Key）可能分布在多个副本节点上，由于网络延迟、节点故障、写入顺序等原因，各节点的数据可能出现不一致**。我们需要一套规模化的验证框架来检测这种不一致。

图片分类优化

现在是使用一个YOLO模型进行分类，要改成先使用一个模型进行大分类，再使用对应大分类的模型进行小分类。

模型文件存在package\ai\data\models\photo-cls目录下

photo-cls-general.pt 模型是一个通用的图片分类模型，用于对图片进行大分类。
如果大类名称为animal则 使用photo-cls-animal.pt模型用于对图片中的动物进行分类。
如果大类名称为person则 使用photo-cls-person.pt 模型用于对图片中的人物进行分类。

如果大类名称对应的模型不存在则直接用大类名称进行分类。
只需要返回一次分类（大分类或小分类）最终的类型
